# /story-packs/ms-102-identity-account-lifecycle-joiner-mover-leaver-symptom-pack.md

Tag: B

## Purpose
This story-pack intentionally creates the most common account lifecycle failures (joiner, mover, leaver) in a Microsoft 365 lab tenant so you can practice your runbook and generate repeatable evidence.

## Symptoms
- New hire cannot access required systems
- User missing licenses or group membership
- Departed employee account still active
- Mailbox or OneDrive still consuming license after termination
- Inconsistent permissions across similar roles
- Manual provisioning causing delays or errors

## Scope
Covers:
- Entra ID user creation and automation
- Group-based licensing
- Dynamic vs assigned group strategy
- Role-based access alignment
- Secure offboarding
- License removal safety
- Mailbox and OneDrive retention handling

Applies to:
- Microsoft 365
- Entra ID
- Exchange Online
- SharePoint / OneDrive
- Intune (only if you have device-bound access requirements configured)

## Preconditions
Roles:
- Global Admin or User Administrator
- License Administrator

Tools:
- Microsoft Graph PowerShell SDK

Standards:
- Established naming and group standards
- Defined department and role mapping model

## Triage checklist gates (do not skip)
Before acting in any real workflow, validate these. For the lab, you still document them as “simulated approved.”
- Confirm request source is authorized (HR or manager)
- Confirm employment status and effective date
- Validate department and role classification
- Validate required license SKU
- Confirm no legal hold or compliance hold applies (for leaver)
- Confirm no shared mailbox or service account dependencies

# Lab setup (one-time)

## Naming convention
Prefix every object with LAB- to make cleanup safe.
- Users: LAB-j-<scenario>, LAB-m-<scenario>, LAB-l-<scenario>
- Groups: LAB-ROLE-<dept>, LAB-LIC-<sku>, LAB-APP-<appname>

## Create baseline groups
Create these Entra ID security groups (Assigned membership):
- LAB-ROLE-Sales
- LAB-ROLE-IT
- LAB-LIC-M365-Sales
- LAB-LIC-M365-IT
- LAB-APP-Sales-Portal

If you use dynamic groups (requires Entra ID P1), also create:
- LAB-DYN-Sales (Dynamic user group rule: (user.department -eq "Sales"))
- LAB-DYN-IT (Dynamic user group rule: (user.department -eq "IT"))

## Group-based licensing baseline
Assign your test SKU via group-based licensing to:
- LAB-LIC-M365-Sales
- LAB-LIC-M365-IT

Important: keep direct user licenses out of this lab unless you are reproducing the manual provisioning symptom.

## Baseline “required system”
Pick one system that can be made assignment-required:
- Enterprise App (recommended): create any test app (Azure AD SAML Toolkit is fine) and set Assignment required = Yes.
- Assign only LAB-APP-Sales-Portal to the app.

## SharePoint permissions baseline
Create a SharePoint site:
- Site: LAB Sales Portal
- Members: LAB-ROLE-Sales

Optional Intune gate (only if you already run this in your tenant):
- Conditional Access that requires compliant device for a target group. If you do not already have this, skip it.

# Scenarios (fault injection + expected signals)

## Scenario 1: New hire cannot access required systems (missing app assignment)
Create user: LAB-j-noapp
- Department: Sales
- Usage location: set
- Add to groups:
  - LAB-ROLE-Sales
  - LAB-LIC-M365-Sales
Intentionally do NOT add:
- LAB-APP-Sales-Portal

Expected “bad” signals:
- My Apps: app missing or access denied
- Enterprise app sign-in: user not assigned (or assignment required failure)

Fix in runbook terms:
- Add user to LAB-APP-Sales-Portal
- Re-test My Apps access

Evidence to collect:
- Screenshot: user group membership showing missing app group
- Screenshot: My Apps missing app or access denied
- Screenshot: after fix, app appears / access works

## Scenario 2: User missing licenses or group membership (license not applied)
Option A (missing license group):
Create user: LAB-j-nolicense
- Department: Sales
- Usage location: set
- Add only to LAB-ROLE-Sales
Intentionally omit:
- LAB-LIC-M365-Sales

Expected “bad” signals:
- User Licenses: no SKU assigned
- Outlook/OneDrive: not available or not provisioned

Fix:
- Add LAB-j-nolicense to LAB-LIC-M365-Sales
- Wait for provisioning, re-test services

Evidence:
- Screenshot: user licenses before fix
- Screenshot: group membership before fix
- Screenshot: user licenses after fix

Option B (license assignment error via missing usage location):
Create user: LAB-j-usage-location-missing
- Department: Sales
- Usage location: blank
- Add to LAB-LIC-M365-Sales

Expected “bad” signals:
- License assignment error on user
- Services not provisioned

Fix:
- Set usage location
- Re-evaluate license status

Evidence:
- Screenshot: license error state
- Screenshot: usage location empty then filled

## Scenario 3: Departed employee account still active (offboarding not executed)
Create user: LAB-l-active
- Department: Sales
- Usage location: set
- Add to:
  - LAB-ROLE-Sales
  - LAB-LIC-M365-Sales
  - LAB-APP-Sales-Portal

Simulate termination date in notes, but do nothing in Entra.

Expected “bad” signals:
- Account enabled
- Sign-in succeeds
- Still in license/app/role groups

Fix (secure offboarding core):
- Block sign-in / disable account
- Remove from role/app groups (or move to “Disabled Users” group if you use that model)
- Start license removal safety checks

Evidence:
- Screenshot: account enabled before
- Screenshot: sign-in still possible (or sign-in logs)
- Screenshot: account disabled after

---

## Scenario 4: Mailbox or OneDrive still consuming license after termination (license group not removed)
Create user: LAB-l-licensed
- Assign via group license (LAB-LIC-M365-Sales)
- Wait for mailbox/OneDrive provisioning

Offboard incorrectly:
- Disable account only
- Do NOT remove from LAB-LIC-M365-Sales

Expected “bad” signals:
- License still assigned
- Mailbox still exists
- OneDrive still has storage and remains tied to licensed state

Fix (license removal safety + retention handling):
- Confirm no legal/compliance hold
- Convert mailbox to shared or set appropriate retention outcome (based on your runbook model)
- Remove from licensing group
- Confirm license returned to pool

Evidence:
- Screenshot: user still in license group after disable
- Screenshot: licenses still assigned
- Screenshot: after removal, license unassigned / mailbox outcome recorded

## Scenario 5: Inconsistent permissions across similar roles (permission drift)
Create users:
- LAB-sales-a
- LAB-sales-b
Both:
- Department: Sales
- Usage location: set
- Add to LAB-ROLE-Sales and LAB-LIC-M365-Sales

On SharePoint site:
- Create folder “LAB Confidential”
- Break inheritance
- Grant access directly to LAB-sales-a only

Expected “bad” signals:
- LAB-sales-a can access folder
- LAB-sales-b gets Access denied despite same role group

Fix:
- Remove direct user permissions
- Re-apply group-based access model (LAB-ROLE-Sales or a dedicated subgroup)
- Re-test as both users

Evidence:
- Screenshot: folder permissions showing direct user assignment
- Screenshot: access denied for peer user
- Screenshot: corrected permissions (group-based)

---

## Scenario 6: Manual provisioning causing delays or errors (process inconsistency)
Run two manual onboarding paths and compare.

Path A (correct order):
1. Create user
2. Set required attributes (department, role, usage location)
3. Add role group
4. Add license group
5. Add app group
6. Validate access

Path B (wrong order and missing steps):
1. Create user
2. Skip usage location
3. Add license group
4. Forget app group
5. Mistype department (example: “Sale”)
6. Validate access and document all mismatches

Expected “bad” signals:
- License errors or delays
- Missing app assignment
- Dynamic group (if used) not matching due to bad department value

Fix:
- Correct attributes
- Correct group memberships
- Standardize the onboarding workflow to eliminate improvisation

Evidence:
- Screenshot: attributes mismatch (department typo)
- Screenshot: group membership mismatch
- Short note: delta between Path A and Path B outcomes

# Verification (pass conditions)
You are done when:
- Joiner scenarios can access required app and SharePoint after fixes
- License errors resolved and services provision correctly
- Leaver scenarios: sign-in blocked, role/app access removed, license reclaimed (unless hold requires otherwise)
- Permission drift eliminated: peers in same role have same effective access
- Manual process errors are eliminated by a consistent workflow

---

# Prevention (what you are proving)
- Group-based licensing only
- Department/role mapping drives group assignment (dynamic or controlled assigned)
- Offboarding includes: sign-in block, group removal, license safety, mailbox/OneDrive retention decision
- Minimal direct permissions, prefer role groups
- Documented checklist prevents HR driven chaos

- Confirm licenses returned
