# Account lifecycle symptom replication (lab tenant)

Purpose: Intentionally create broken joiner, mover, leaver states so you can practice triage and validate fixes against your canonical runbook: /runbooks/ms-102/identity-account-lifecycle-automation.md.

## Symptoms

- New hire cannot access required systems
- User missing licenses or group membership
- Departed employee account still active
- Mailbox or OneDrive still consuming license after termination
- Inconsistent permissions across similar roles
- Manual provisioning causing delays or errors

## Scope

Creates common failure modes in:
- Entra ID users, groups, and app assignments
- Group-based licensing
- SharePoint permissions
- Exchange Online mailbox presence (as a license signal)

Non-goals:
- Hybrid identity (AD Connect)
- Intune device compliance gates (optional add-on only)

## Preconditions

Roles (use least privilege if you want realism):
- User Administrator (user create/disable)
- Groups Administrator (group membership)
- License Administrator (license assignment)
- Exchange Administrator (mailbox checks)
- SharePoint Administrator (site + permissions)

Tenant/licensing:
- At least 2 test user licenses that include Exchange and OneDrive (ex: M365 Business Premium or M365 E3)
- If you want dynamic groups: Entra ID P1

Safety:
- Use a dedicated test tenant or a clearly isolated lab namespace (prefix everything with LAB-)
- Never reuse real people names or real emails

## Triage checklist (before you start breaking things)

1. Confirm you can access:
   - Entra admin center
   - Microsoft 365 admin center
   - Exchange admin center
   - SharePoint admin center
2. Confirm at least one test SKU is available (not fully consumed).
3. Decide if you will use dynamic groups (requires P1). If not, do assigned groups only.

## Fix steps (how to replicate each symptom)

### Step 0: Create the baseline objects (do this once)

1. Create groups (Security groups, Assigned membership):
   - LAB-ROLE-Sales
   - LAB-LIC-M365-Sales
   - LAB-APP-Sales-Portal

2. Assign licenses to the licensing group:
   - Entra admin center -> Billing -> Licenses -> All products -> select your SKU -> Licensed groups -> Assign -> choose LAB-LIC-M365-Sales

3. Create an Enterprise App that requires assignment (a safe dummy):
   - Entra admin center -> Enterprise applications -> New application
   - Search gallery for "Azure AD SAML Toolkit" (or any simple SAML test app)
   - Create it
   - In the app: Properties -> Assignment required = Yes
   - Users and groups -> Add user/group -> add LAB-APP-Sales-Portal

4. Create a SharePoint site for permission testing:
   - SharePoint admin center -> Sites -> Active sites -> Create
   - Name: LAB Sales Portal
   - Add LAB-ROLE-Sales as Members (or create a SharePoint group and add LAB-ROLE-Sales)

### Symptom 1: New hire cannot access required systems (missing app assignment group)

1. Create user: LAB-joiner-noapp
   - Set Usage location
   - Set Department = Sales
2. Add user to:
   - LAB-LIC-M365-Sales (so they are licensed)
   - LAB-ROLE-Sales (so they look "correct")
3. Intentionally do NOT add user to:
   - LAB-APP-Sales-Portal
4. Verify symptom:
   - Sign in as LAB-joiner-noapp
   - Open My Apps (myapps.microsoft.com)
   - The app will be missing or access denied because assignment is required.

### Symptom 2: User missing licenses or group membership (license not assigned or license fails)

Option A (simplest):
1. Create user: LAB-joiner-nolicense
2. Add to LAB-ROLE-Sales only.
3. Do NOT add to LAB-LIC-M365-Sales.
4. Verify symptom:
   - Sign in as LAB-joiner-nolicense
   - Outlook on the web will fail (no mailbox/license) and OneDrive will fail.

Option B (common real-world mistake):
1. Create user: LAB-joiner-usage-location-missing
2. Leave Usage location blank.
3. Add user to LAB-LIC-M365-Sales.
4. Verify symptom:
   - Entra user -> Licenses shows an assignment error (varies by SKU/tenant).
   - Services (Outlook/OneDrive) do not provision.

### Symptom 3: Departed employee account still active (offboarding not executed)

1. Create user: LAB-leaver-active
2. Add to:
   - LAB-LIC-M365-Sales
   - LAB-ROLE-Sales
   - LAB-APP-Sales-Portal
3. "Terminate" them in your notes, but intentionally do nothing in Entra.
4. Verify symptom:
   - Sign in still works.
   - They can still access My Apps and the SharePoint site.

### Symptom 4: Mailbox or OneDrive still consuming license after termination (license not removed)

1. Create user: LAB-leaver-licensed
2. Add to LAB-LIC-M365-Sales and wait 10 to 30 minutes for provisioning.
3. Terminate them but only do:
   - Entra user -> Block sign-in (or Account enabled = No)
4. Intentionally keep them in LAB-LIC-M365-Sales (or leave a direct license assigned).
5. Verify symptom:
   - Microsoft 365 admin center -> Users -> Active users -> user -> Licenses and apps still shows license assigned.
   - Exchange admin center -> Recipients -> Mailboxes still shows mailbox exists.
   - OneDrive storage still exists and remains tied to the licensed user object.

### Symptom 5: Inconsistent permissions across similar roles (same role, different effective access)

1. Create two users:
   - LAB-sales-a
   - LAB-sales-b
2. Add both to:
   - LAB-LIC-M365-Sales
   - LAB-ROLE-Sales
3. On the SharePoint site, create a folder "LAB Confidential" and break inheritance:
   - Grant access ONLY to LAB-sales-a (direct user permission)
   - Do NOT grant to LAB-ROLE-Sales
4. Verify symptom:
   - LAB-sales-a can open folder
   - LAB-sales-b gets Access denied
This reproduces "same role on paper, different permissions in reality."

### Symptom 6: Manual provisioning causing delays or errors (inconsistent process)

Run this twice and compare outcomes.

Manual path A (correct order):
1. Create user -> set attributes (Department, Job Title, Usage location)
2. Add to role group(s)
3. Add to licensing group(s)
4. Add to app assignment group(s)
5. Validate access

Manual path B (wrong order, common mistakes):
1. Create user -> do NOT set Usage location
2. Add to licensing group
3. Forget app group
4. Later set Department incorrectly (ex: "Sale" instead of "Sales")
5. Validate access and capture the mismatches:
   - License errors or delays
   - Missing app assignment
   - Group logic not matching intended role

## Verification (what "broken" looks like)

- Missing app access: app absent from My Apps or access denied
- Missing license: Outlook/OneDrive not provisioned, license not present in user blade
- Leaver active: sign-in succeeds after "termination date"
- Leaver still consuming license: license assigned while account disabled, mailbox still present
- Inconsistent permissions: one peer can access a site/folder/app, the other cannot
- Manual errors: user object attributes and group membership do not match the role model

## Prevention (what the real fix should enforce)

- Group-based licensing only, no direct licenses
- Standard role groups, no direct user permissions to sites/apps unless formally approved
- Offboarding checklist that always includes: block sign-in, remove role/app groups, license removal safety checks, mailbox/OneDrive handling decisions
- Attribute hygiene (Department, Job Title, Usage location) before any automation depends on them
- One documented workflow, no improvisation

## Rollback (cleanup)

- Remove all LAB- users
- Remove all LAB- groups
- Remove the test enterprise application
- Delete the LAB SharePoint site
- Confirm licenses returned to pool
