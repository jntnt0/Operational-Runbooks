# /story-packs/ms-102-user-cant-sign-in-root-cause-tree-symptom-pack.md

Tag: B

## Purpose
Reproduce common interactive sign-in failures in a lab tenant so you can walk the runbook root-cause tree, prove signals in sign-in logs, apply the fix, and capture evidence.

## Symptoms
- User gets “Your account or password is incorrect”
- User gets “Your sign-in was blocked” or “You can’t get there from here”
- MFA loops or fails
- Login works on one device but not another
- Sign-in succeeds but app access fails (Outlook, Teams, SharePoint)
- New user cannot sign in at all

## Scope
Covers interactive user sign-in to:
- Microsoft 365 (portal.office.com)
- Exchange Online (Outlook desktop/mobile/OWA)
- Teams (desktop/mobile/web)
- SharePoint / OneDrive
- Azure portal (if user is blocked there)

Excludes:
- Service principals and app-only auth
- On-prem AD authentication outages beyond basic checks

## Preconditions
Access:
- Entra admin center
- Ability to view Sign-in logs and Audit logs (licensing affects detail)
- Ability to check Conditional Access policies

Tools (optional):
- Microsoft Graph PowerShell SDK

Minimum roles (least privilege):
- Global Reader (read-only triage)
- Security Reader (view sign-in details)
- Conditional Access Administrator (CA policy review)
- User Administrator / Helpdesk Administrator (reset password, unblock user)

## Triage checklist gates (document these for every scenario)
Collect first, then change things.
- Who is affected (single user, group, tenant-wide)?
- What app is failing (M365 portal, Teams, Outlook, Azure portal)?
- What device and network (corp LAN, home, mobile, VPN)?
- When did it start (exact timestamp, timezone)?
- What error is shown (screenshot, error code, correlation ID)?
- Has anything changed (password reset, device replaced, CA policy change, license change)?
- Check Service Health for current incidents.

# One-time lab setup

## Safety rules (do not skip)
- Use a dedicated lab tenant or LAB- prefixed objects only.
- Do not apply Conditional Access to your admin account.
- Keep a break-glass style admin excluded from all CA policies you create in this story-pack.

## Create test users
Create these cloud-only users (set Usage location for each):
- LAB-sso-good
- LAB-sso-badpass
- LAB-sso-blocked
- LAB-sso-mfa
- LAB-sso-client-diff
- LAB-sso-appfail
- LAB-sso-newuser

Recommended settings:
- Set a known temporary password for each.
- For LAB-sso-newuser, set “Require password change at next sign-in” (for one of the scenarios below).

## Create groups
Security groups (Assigned membership):
- LAB-CA-Block
- LAB-CA-RequireMFA
- LAB-CA-BlockDesktopMobile
- LAB-LIC-M365-BASIC
- LAB-APP-AssignmentRequired

Add users to groups like this (baseline, before fault injection):
- LAB-sso-blocked -> LAB-CA-Block
- LAB-sso-mfa -> LAB-CA-RequireMFA
- LAB-sso-client-diff -> LAB-CA-BlockDesktopMobile
- LAB-sso-appfail -> LAB-APP-AssignmentRequired
- Everyone except LAB-sso-appfail -> LAB-LIC-M365-BASIC (leave LAB-sso-appfail unlicensed on purpose)

## Conditional Access policies
Create these policies (exclude your admin/break-glass account from each):

Policy 1: Block sign-in
- Users: LAB-CA-Block
- Cloud apps: All cloud apps
- Grant: Block access

Policy 2: Require MFA
- Users: LAB-CA-RequireMFA
- Cloud apps: All cloud apps (or Microsoft 365 if you want narrower)
- Grant: Require multifactor authentication

Policy 3: “Works on browser, fails on desktop/mobile” (no Intune required)
- Users: LAB-CA-BlockDesktopMobile
- Cloud apps: Microsoft 365 (or All cloud apps)
- Conditions: Client apps -> select Mobile apps and desktop clients
- Grant: Block access

## “Sign-in succeeds but app access fails” target
Pick one of these (do either, both are valid evidence):

Option A (license based):
- Leave LAB-sso-appfail out of LAB-LIC-M365-BASIC.
- This user can sign in, but fails when using licensed apps (Teams/Outlook/SharePoint depending on SKU).

Option B (enterprise app assignment required):
- Create a test Enterprise App (gallery app is fine).
- Set “Assignment required” to Yes.
- Assign only LAB-APP-AssignmentRequired group.
- Test access as a user not assigned to the app to get a clean “signed in, no app access” failure.

# Scenarios (fault injection, signals, fix, evidence)

## Scenario 1: “Your account or password is incorrect”
Target symptom:
- User gets “Your account or password is incorrect”

Fault injection:
- Attempt sign-in as LAB-sso-badpass using the wrong password 3 times.

Expected signals (before any fixes):
- Sign-in logs show failed sign-in for the user.
- Error text reflects invalid credentials.

Fix actions:
- Reset password for LAB-sso-badpass.
- Confirm sign-in succeeds with the new password.

Evidence to collect:
- Screenshot: end-user error message
- Screenshot: sign-in log entry for the failure (include timestamp, app, status)
- Screenshot: sign-in log entry showing success after reset

---

## Scenario 2: “Your sign-in was blocked” / “You can’t get there from here”
Target symptom:
- User gets blocked message

Fault injection:
- LAB-sso-blocked is already in LAB-CA-Block.
- Attempt sign-in to portal.office.com.

Expected signals:
- Sign-in logs show failure with Conditional Access as the control point.
- Conditional Access tab shows policy “Block sign-in” applied.

Fix actions:
- Remove LAB-sso-blocked from LAB-CA-Block (or disable the policy temporarily).
- Re-test sign-in.

Evidence:
- Screenshot: end-user blocked message
- Screenshot: sign-in log details showing Conditional Access decision
- Screenshot: group membership change (before/after)
- Screenshot: successful sign-in after fix

---

## Scenario 3: MFA loops or MFA fails
Target symptom:
- MFA loops or fails

Fault injection (choose one, keep it simple):

Option A: Require MFA but user has no methods registered
- Ensure LAB-sso-mfa is in LAB-CA-RequireMFA.
- Remove registered authentication methods for LAB-sso-mfa (Authentication methods blade).
- Attempt sign-in.

Option B: Stale sessions and repeated prompts
- Require MFA (policy already exists).
- Revoke user sessions for LAB-sso-mfa.
- Attempt sign-in repeatedly across browser and Teams web to induce repeated prompts.

Expected signals:
- Sign-in logs show MFA requirement, and the failure/prompt stage depending on the option you chose.
- Authentication details show MFA step not satisfied or repeatedly required.

Fix actions:
- Re-register a valid method for LAB-sso-mfa (Authenticator is the easiest).
- Confirm sign-in works with a clean MFA prompt.

Evidence:
- Screenshot: authentication methods state (before/after)
- Screenshot: sign-in log authentication details showing MFA required and then satisfied
- Screenshot: end-user MFA screen (no secrets shown)

---

## Scenario 4: Login works on one device but not another (client type difference)
Target symptom:
- Login works on one device but not another

Fault injection:
- LAB-sso-client-diff is in LAB-CA-BlockDesktopMobile.
- Test from:
  - Browser sign-in to portal.office.com (should work)
  - Outlook desktop or Teams desktop sign-in (should fail)

Expected signals:
- Browser sign-in success.
- Desktop/mobile client sign-in blocked by Conditional Access (client apps condition).

Fix actions:
- Adjust policy scope (remove the user, or change client app targeting).
- Re-test Outlook or Teams sign-in.

Evidence:
- Screenshot: successful browser sign-in
- Screenshot: failed desktop/mobile sign-in experience
- Screenshot: sign-in logs showing different client app types and different outcomes

---

## Scenario 5: Sign-in succeeds but app access fails (Outlook, Teams, SharePoint)
Target symptom:
- Sign-in succeeds but the app fails

Fault injection (pick one based on what you set up):
Option A (license based):
- LAB-sso-appfail is not in LAB-LIC-M365-BASIC.
- Sign in to portal.office.com (should succeed).
- Attempt to open Teams or Outlook on the web (should fail due to licensing).

Option B (enterprise app assignment required):
- Sign in as a user not assigned to the enterprise app.
- Attempt to launch the app from My Apps.
- Expect access denied due to missing assignment.

Expected signals:
- Sign-in logs may show success for initial auth.
- App access failure shows licensing or assignment issue (not an auth failure).

Fix actions:
- License path: add user to LAB-LIC-M365-BASIC, wait for provisioning, re-test.
- Assignment path: add user/group assignment to the enterprise app, re-test.

Evidence:
- Screenshot: sign-in success
- Screenshot: Teams/Outlook/SharePoint error (or app assignment error)
- Screenshot: license tab or enterprise app assignment showing the fix

---

## Scenario 6: New user cannot sign in at all
Target symptom:
- New user cannot sign in at all

Fault injection (do both, they cover common real causes):
A) Account disabled by mistake
- Set LAB-sso-newuser “Block sign-in” = Yes.
- Attempt sign-in.

B) Password change required but user tries a client that does not handle it cleanly
- Set “Require password change at next sign-in” for LAB-sso-newuser.
- Try sign-in via a desktop client first (if available), then browser.

Expected signals:
- A) Sign-in logs show the account is blocked/disabled.
- B) Sign-in may fail or loop until password change is completed in a supported flow.

Fix actions:
- A) Re-enable sign-in.
- B) Complete password change via browser, then retry the client.

Evidence:
- Screenshot: blocked sign-in toggle state
- Screenshot: sign-in logs for the failure
- Screenshot: successful sign-in after enabling / completing password change

# Verification (pass conditions)
You are done when:
- Each symptom is reproduced on demand.
- For each scenario, you have at least one sign-in log entry that matches the failure type.
- Each scenario is fixed and re-tested successfully.
- You captured the same evidence set every time (repeatable).

# Prevention notes (what the runbook should drive)
- Always collect correlation ID, exact time, app name, and device/network before changing anything.
- Use Conditional Access scoping discipline (small test group first).
- Prefer group-based licensing to avoid random app failures.
- Standardize onboarding steps to prevent “new user cannot sign in” mistakes.

# Rollback / Cleanup
- Delete LAB- users and groups.
- Disable and delete the Conditional Access policies created for this story-pack.
- Remove the test enterprise app (if created).
- Confirm licenses returned to the pool.
